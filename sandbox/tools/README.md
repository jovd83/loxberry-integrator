# LoxBerry sandbox

Two ways to run a LoxBerry sandbox so plugins generated by this skill
(e.g. `sandbox/Marstek-cloud`) can be installed and tested end-to-end.

| | Docker container *(recommended)* | VirtualBox VM *(deprecated)* |
|---|---|---|
| Setup time | seconds | hours (DietPi + LoxBerry installer in the VM) |
| Persistence | named Docker volumes survive `compose down` | full VM disk |
| Auth needed | none (no UAC) | admin (UAC for VirtualBox install + Mount-DiskImage) |
| Reliability | works first try on Windows 11 + Docker Desktop | broke on UEFI image / WSL2-kernel partition scan |
| Realism | 95% — same Apache+systemd+mosquitto stack as a Pi | 100% — real LoxBerry 3.x install |

**Use Docker** unless you need to validate something that needs a real
networked LoxBerry host on your LAN (e.g. Loxone Miniserver auto-discovery).

---

## Docker workflow (recommended)

Container: `loxberry-sandbox` based on `boernmasta/loxberry:latest` (DietPi +
LoxBerry 3.0.1.3, amd64, ~1.1 GB).

### One-time setup

Docker Desktop must be running.

```bash
cd c:/projects/skills/loxberry-integrator/sandbox/tools
docker compose pull          # first pull is ~1.1 GB
docker compose up -d         # starts the container (named volumes auto-created)
```

### Daily use

```bash
docker compose stop          # pause (state persists in named volumes)
docker compose start         # resume
docker compose logs -f       # tail container logs
docker compose down          # remove the container (volumes stay)
docker compose down -v       # WIPE everything including config/plugins/logs
```

### Open the UI

| URL | Login | Notes |
|---|---|---|
| `http://localhost:18080` | `loxberry` / `loxberry` | Web UI (HTTP Basic Auth) |
| `mqtt://localhost:1883` | anonymous (default) | Mosquitto broker (pre-installed) |

The plugin install **SecurePIN** in this sandbox is force-set to **`1234`**
on first install (see *Install marstek-cloud below*). On a real LoxBerry you
set this via the web UI in Settings &rarr; Admin Access &rarr; SecurePIN.

### Install marstek-cloud in the sandbox

The freshly-built ZIP is bind-mounted at `/incoming/marstek-cloud-0.1.0.zip`
inside the container.

**Web UI:** *Plugin Management* &rarr; *Plugin install* &rarr; upload the ZIP
from your host: `c:\projects\skills\loxberry-integrator\sandbox\marstek-cloud-0.1.0.zip`,
enter the SecurePIN (`1234` by default in the sandbox), click *Install*.

**Or CLI** (fully scripted, idempotent):

```bash
# Set the SecurePIN to a known value the first time (Windows host, Git Bash):
MSYS_NO_PATHCONV=1 docker exec loxberry-sandbox bash -c '
  perl -e "print crypt(q{1234}, q{IG})" > /opt/loxberry/config/system/securepin.dat &&
  chown loxberry:loxberry /opt/loxberry/config/system/securepin.dat &&
  chmod 0600 /opt/loxberry/config/system/securepin.dat
'

# Install (or re-install on top — installer handles the upgrade)
MSYS_NO_PATHCONV=1 docker exec loxberry-sandbox bash -c '
  export PERL5LIB=/opt/loxberry/libs/perllib
  /opt/loxberry/sbin/plugininstall.pl \
      action=install \
      file=/incoming/marstek-cloud-0.1.0.zip \
      pin=1234
  chmod 755 /opt/loxberry/webfrontend/htmlauth/plugins/marstek-cloud/index.cgi
'
```

Open `http://localhost:18080/admin/plugins/marstek-cloud/index.cgi` — the
plugin page renders inside the LoxBerry UI shell.

### Iterating on the plugin source

Whenever you edit anything under `sandbox/Marstek-cloud/`:

```bash
# 1. rebuild the install ZIP
cd c:/projects/skills/loxberry-integrator/sandbox
rm -f marstek-cloud-0.1.0.zip
mkdir -p _zipstage/marstek-cloud
cd Marstek-cloud
find . -type d -name __pycache__ -prune -o -type f ! -name '*.pyc' -print |
  while read f; do
    mkdir -p "../_zipstage/marstek-cloud/$(dirname "$f")" &&
    cp "$f" "../_zipstage/marstek-cloud/$f"
  done
cd ../_zipstage
python -m zipfile -c ../marstek-cloud-0.1.0.zip marstek-cloud
cd ..
rm -rf _zipstage

# 2. re-install in the sandbox (the install command above is idempotent —
#    LoxBerry replaces the previous version cleanly)
```

### Useful container probes

```bash
# Tail the daemon log (after the plugin is enabled and configured)
MSYS_NO_PATHCONV=1 docker exec loxberry-sandbox \
  tail -f /opt/loxberry/log/plugins/marstek-cloud/marstek-cloud.log

# Subscribe to all MQTT traffic from the host
docker exec loxberry-sandbox mosquitto_sub -h localhost -t '#' -v

# Drop into a shell inside the container
MSYS_NO_PATHCONV=1 docker exec -it loxberry-sandbox bash
```

### Caveats

- This image runs systemd in Docker, which needs `--privileged` and
  `cgroup: host` on Docker Desktop / WSL2 (compose already sets these).
- `Mount-DiskImage`-based plugin install isn't relevant here — files come in
  via the bind-mount.
- Some plugins that depend on real hardware (GPIO, USB, 1-Wire) won't work
  in a container. The Marstek Cloud plugin is pure network and is unaffected.
- The container has its own mosquitto broker on `localhost:1883`. If you
  already run a broker on your host on that port, change the host-side port
  in `docker-compose.yml`.

## VirtualBox workflow (deprecated)

Kept around for reference only. The script `build-loxberry-vm.ps1` plus
`Automation_Custom_Script.sh` will, in principle:

1. Install VirtualBox + 7-Zip via winget.
2. Download the DietPi UEFI x86_64 OVA, decompress, convert to raw, patch
   the ext4 root with a `dietpi.txt` preseed via a privileged debian
   container, convert back to VMDK, register a VM, boot it.
3. Wait up to 3 h for the in-VM LoxBerry installer to finish.

In practice this failed for us at the kernel-partition-scan step inside
Docker Desktop's WSL2 kernel — `losetup -P` doesn't expose `*p1`/`*p2`
device nodes. The script was rewritten to mount by byte offset (which
works), but the overall 2-hour critical path makes the Docker workflow
above strictly better for plugin iteration.

If you genuinely need a real LoxBerry VM (e.g. for Miniserver
auto-discovery), either fix the remaining bits of `build-loxberry-vm.ps1`
or install LoxBerry by hand on a Raspberry Pi / x86 box per the official
wiki: <https://wiki.loxberry.de/>.
